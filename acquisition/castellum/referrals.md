# Castellum referrals, communities, and backlinks

Research date: **2026-07-17**

Scope: **public links and mentions that can be tied to the email product at
`castellummail.com`**

This file separates observed exposure from inferred traffic and conversion.

## Executive finding

No credible public referral source was found.

DataForSEO indexed two referring pages, but both are obvious nofollow
SEO-spam pages from one IP. Exact-domain Content Analysis returned zero
mentions. Narrow public searches did not surface a Castellum Mail launch,
review, founder promotion, community thread, partnership, or directory
listing.

The public GitHub release repository is a delivery surface, not evidence of
where awareness came from. It had zero stars, forks, and open issues while its
disk images had 24 requests.

There is no public source-to-download, source-to-install, source-to-mailbox,
or source-to-retained-use attribution.

[`joanmarcriera/castellum-releases`](https://github.com/joanmarcriera/castellum-releases)

| Disk-image requests | 24 |

The source is private; the repository hosts the site and release assets. That
limits open-source discovery and collaboration by design.

The site says the in-app `Report a Problem` action attaches diagnostics and
opens a prefilled GitHub issue. A public workflow labels every new issue
`needs-triage` and adds `from-app` when the body contains diagnostic markers.
No public issue was open at capture.

Zero issues can mean:

- nobody submitted one;
- reports were closed or removed;
- people contacted the maker another way;
- users never installed or reached the reporter;
- the product had no reportable problem.

It cannot distinguish among those cases.

## What appears to matter

The public evidence currently stops at owned distribution:

```text
unknown awareness
→ owned homepage
→ owned release repository
→ 24 unqualified disk-image requests
```

That sequence cannot identify what message, person, or channel caused a
request. It also cannot tell whether a request became:

- a clean launch;
- a Gmail or IMAP connection;
- a local-AI setup;
- useful mail work;
- a second session;
- a retained early-access user.

The low-quality backlink rows are less informative than the absence of a
credible human recommendation. A founder with an existing professional
network could create one, but that experiment is not publicly observed.

## Tecotype implications

- Treat GitHub or CDN asset requests as a distribution diagnostic, not users.
- Exclude automated link spam before reporting referral reach.
- A technically credible founder identity can support a future launch story,
  but only a source-tagged path can show acquisition.
- Do not place broad community or reviewer outreach ahead of clean first run,
  provider authorization, signed distribution, safe sending, and activation
  measurement.
- When Tecotype is ready, use one disclosed founder/reviewer source at a time
  and preserve:

```text
source
→ message
→ landing page
→ signed install
→ provider connection
→ message-matched first value
→ voluntary return
```

- Do not buy links, seed undisclosed endorsements, or use publication/profile
  reach as a proxy for product use.
