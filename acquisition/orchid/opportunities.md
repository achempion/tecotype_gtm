# Orchid-informed marketing and distribution opportunities

This file applies Orchid's public acquisition evidence to Tecotype's
marketing and distribution only. It assumes the published Tecotype offer can
support macOS, Windows, and Linux and that product delivery is handled
elsewhere. Campaign copy must still match the platforms and capabilities
actually public at the time. Mobile is out of scope.

## Transferable finding

Orchid's Zero era shows that a coordinated technical launch can create a
durable branded search result page. Its transition also shows the cost of
breaking continuity between:

```text
message
→ category
→ domain
→ CTA
→ activation
→ payment
```

Tecotype should seek the first effect while avoiding the second.

## Experiment 1: one-message launch bundle

Use one precise desktop-email promise across a small launch bundle:

```text
public proof page
+ one high-fit community contribution
+ one launch surface
+ one founder explanation
→ one canonical Tecotype page
→ attributed desktop activation
→ payment
```

Candidate messages to test separately:

- **Find mail the way you remember it**, demonstrated with one multilingual
  retrieval;
- **Calm control over your mail**, demonstrated through one complete triage
  sequence; or
- **Keyboard-first mail without the performance theatre**, demonstrated with
  an inspectable workflow.

Do not combine all three in the headline.

| Field | Design |
| --- | --- |
| Audience | One validated email-heavy professional segment |
| Sources | One relevant community and one launch surface, each with a unique source token |
| Primary metric | Paid users per qualified landing visit |
| Diagnostic sequence | Visit → evaluation start → account connected → lead job completed → voluntary return → payment |
| Continue signal | At least one source produces paid users at acceptable effort after a minimum useful cohort |
| Stop or revise | High attention with weak qualified activation, or comments show category misunderstanding |

Product Hunt points, post votes, impressions, and stars are diagnostics only.

## Experiment 2: a permanent technical proof surface

Zero's repository created search results, backlinks, trust discussion, and
community participation. Tecotype does not need to open its whole codebase to
borrow the distribution pattern.

Publish a permanent, indexable proof surface containing:

- the exact local-mailbox boundary;
- what Tecotype infrastructure does and does not receive;
- a diagram of provider communication;
- an inspectable Better-search example;
- platform-specific availability;
- a public changelog;
- a security contact; and
- links to the canonical evaluation path.

Distribute it to developer, privacy, and keyboard-workflow audiences only when
it directly answers their evaluation question.

| Metric | Role |
| --- | --- |
| Qualified inbound links | Discovery diagnostic |
| Return visits to proof | Evaluation diagnostic |
| Proof-page to activation | Primary behavior |
| Activation to payment | Commercial decision |

Repository stars or document views do not substitute for the last two.

## Experiment 3: preserve source and category continuity

Before any launch, verify that:

- title, description, canonical URL, social card, headline, and CTA all name
  the same desktop-email category;
- every external campaign points to the same canonical job page;
- source data survives any redirect and the desktop handoff;
- platform-specific pages disclose availability accurately; and
- old campaign URLs remain useful rather than becoming generic redirects.

Run a monthly audit:

```text
old public URL
→ current matching message
→ live CTA
→ source token preserved
→ paid outcome attributable
```

This is the direct response to the Zero-to-Orchid fragmentation.

## Experiment 4: demonstration-led editorial

Orchid's strongest current content is a concrete outcome story. Tecotype can
build a narrower evidence library:

1. one canonical Better-search demonstration;
2. one keyboard triage walkthrough;
3. one customer story tied to a recent, specific email job; and
4. one privacy architecture page.

Each page should target a distinct evaluation question and lead to the same
attributed funnel. Do not build a broad blog until at least one page produces
qualified activations and paid users.

## Experiment 5: channel contrast, not channel aggregation

Zero's Product Hunt and Hacker News launches showed dramatically different
visible engagement. Tecotype should not publish identical copy everywhere and
sum the reactions.

For each source, record:

| Source | Audience question | Message variant | Qualified visits | Activated | Paid | Renewed |
| --- | --- | --- | ---: | ---: | ---: | ---: |
| Launch site | New-product evaluation | Chosen lead promise |  |  |  |  |
| Technical community | Trust and implementation | Proof surface |  |  |  |  |
| Workflow community | Specific email job | Demonstration |  |  |  |  |

Continue a source only when it contributes paid users or materially improves
an evidenced conversion step.

## What not to copy

- Counting a waitlist or early-access form as adoption.
- Using open-source popularity as evidence of hosted-product retention.
- Broad “AI assistant” language that removes Tecotype from the desktop-email
  buying decision.
- Mixing old and new brands without a source-preserving transition.
- A primary CTA that loses source data at an external protocol handoff.
- Launch discounts that manufacture urgency.
- Repeating the same promotional post across communities without adapting it
  to a real audience question.
- Building integrations, editorial inventory, or community infrastructure for
  backlinks alone.

## Measurement standard

Tecotype distribution should be counted only as:

```text
permanent public discovery
→ attributable qualified visit
→ attributed activation
→ payment
→ retained paid use
```

Everything before activation is evidence about discovery and evaluation, not
a completed distribution channel.
